이번 주의 나는 [[로컬 LLM]]을 업무환경 안으로 끌어들이는 문제에서 출발했다. [[LM Studio]]에서 불러온 모델을 [[Cursor]]의 Agent처럼 쓰고 싶었고, 메뉴 위치와 접속 확인 방법을 계속 물었다. 특히 내부망 주소와 로컬 서버 포트를 확인하면서, [[AI Agent]]가 클라우드 도구인지 로컬 도구인지에 따라 접근 가능한 자원이 달라진다는 점을 체감했다. [[사설망]] IP, 프록시, 유료 플랜 가능성, 로컬 API 엔드포인트 확인 같은 질문은 결국 “내 업무 데이터를 어디까지 안전하게, 어디까지 자동화할 수 있는가”라는 문제로 이어졌다.

이 흐름은 곧 [[Windows 권한]]과 환경설정 문제로 내려왔다. [[hosts 파일]]을 시스템 경로에 강제로 덮어쓰는 방법, [[cmd]]에서 프로세스를 종료하는 방법, PID 확인과 `taskkill` 사용 같은 질문은 겉으로는 작은 운영체제 팁처럼 보인다. 그러나 실제로는 [[로컬 LLM]], [[Cursor]], [[감정분석 GUI]], [[SAP 자동화]]를 안정적으로 돌리기 위해 환경을 통제하려는 시도였다. 나는 도구를 설치하는 데서 멈추지 않고, 충돌하는 프로세스와 권한, 네트워크 경로까지 직접 다루려 했다.

[[KOTE]]와 [[감정분석]]은 이번 주의 가장 선명한 기술 축이었다. [[Hugging Face]]의 [[KoELECTRA]] 기반 모델을 [[Python]]에서 불러오는 코드, 실제 `kote_lightning.bin` 파일을 로딩하는 방식, 그리고 [[PyInstaller]]로 모델 파일까지 포함해 실행 파일로 묶는 방법을 물었다. 이어서 GUI 요구사항도 구체화했다. 모델 파일을 사용자가 직접 로딩하고, 직접 입력 텍스트와 파일 선택 분석을 나누며, 원본 데이터 전체에 감정분석 결과 열을 추가하는 구조를 원했다. 나는 감정분석을 노트북 예제에서 끝내지 않고, 비전문 사용자도 반복 실행할 수 있는 업무도구로 만들고자 했다.

분석 관점도 깊어졌다. 단순히 긍정·중립·부정을 나누는 데 그치지 않고, 세분화된 감정 강도 데이터가 충분히 있을 때 [[지도학습]]으로 무엇을 더 할 수 있는지 물었다. 여기서 내가 떠올린 방향은 [[구조적 주제모델링]]과 결합해 감정의 극성 변화와 응답 주제 변화를 함께 보는 것이었다. 이 질문은 [[감정분석]]을 점수표로만 보지 않고, 사람을 더 깊이 이해하기 위한 [[데이터 분석]] 체계로 확장하려는 태도를 보여준다. 감정, 주제, 응답자 속성, 시간 변화가 연결될 때 교육 피드백은 단순 만족도보다 훨씬 풍부한 해석 자료가 된다.

다른 분석 질문도 같은 결을 가졌다. [[다단계 그룹화]]와 [[빈도분석]]에서는 로데이터와 분석테이블을 함께 보며 위계적 집계의 원리와 효율적 방법을 물었다. [[언어학]]에서는 음성, 음운, 분절음운, 자음, 모음, 비분절음운, 길이, 강세, 형태소, 최소대립쌍을 학문적 위계로 다시 정리하려 했다. 주제는 달랐지만 방식은 같았다. 나는 흩어진 개념을 표면적으로 암기하기보다, 분류체계와 위계를 세워 다시 사용할 수 있는 지식 구조로 바꾸려 했다.

자동화 측면에서는 [[Git Bash]], [[PowerShell]], [[GitHub]]가 함께 등장했다. 특정 폴더 전체를 커밋하고 푸시하는 스크립트를 요청했고, 같은 작업을 PowerShell에서 수행하는 방법도 다시 물었다. 이는 [[감정분석]] 프로젝트를 단발성 파일 묶음이 아니라 버전 관리 가능한 산출물로 전환하려는 움직임이었다. 코드와 모델, GUI와 데이터가 계속 바뀌는 상황에서는 저장소에 올리고 이력을 남기는 습관 자체가 작업 품질을 지켜 준다.

[[SAP GUI 스크립팅]] 질문은 다시 업무자동화의 심화로 이어졌다. 특정 [[T-code]]에 존재하는 기술정보, [[ALV]] 정보, GUI Scripting 정보를 모두 가져오는 [[GitHub]] 리포지토리를 찾고 싶어 했고, 이미 작성 중인 다운로드 자동화 코드의 구조도 공유했다. 나는 SAP 자동화를 버튼 클릭 반복이 아니라, 화면의 기술정보와 테이블 구조를 이해하는 방식으로 확장하려 했다. [[SAP 자동화]]와 [[Python 자동화]]가 결합될 때, 업무 담당자의 반복 작업은 재현 가능한 데이터 파이프라인으로 바뀔 수 있다.

후반에는 [[박사과정]] 진학 관련 문서를 다듬었다. 지도교수님께 진학 계획과 근황을 알리고, 별도 준비사항과 면담 가능 여부를 여쭙는 메일을 구성했다. 구체적인 개인정보와 수신자 정보는 공개 글에 남기지 않지만, 이 대화의 핵심은 분명하다. 나는 [[생성형 AI]]를 활용한 교육체계와 프로그램 개발·운영 경험을 [[산업인력개발학과]] 박사과정의 문제의식과 연결해 설명하려 했다. 업무 현장의 [[HRD]] 실천과 연구자로서의 진학 준비가 같은 축 위에 놓이기 시작했다.

생활과 문서 영역도 함께 있었다. 유아교육 놀이계획안을 비판적으로 다듬고, 짧은 안내 문구를 더 간결하게 바꾸며, 보상 안내 요청 문장을 [[일본어 번역]]과 한국어 번역으로 정리했다. 민감한 사고·보상 맥락의 세부사항은 남기지 않는다. 대신 이번 주의 패턴만 남긴다. 나는 기술 도구, 연구 준비, 업무 자동화, 가족·생활 문서를 모두 같은 방식으로 다뤘다. 원문을 구조화하고, 실행 가능한 문장으로 바꾸고, 반복 가능한 도구나 절차로 남기는 방식이었다.
